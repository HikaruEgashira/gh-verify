# ADR-0002: 設計公理 (Design Principals)

## ステータス
採択

## コンテキスト
ADR-0001 で Control/Evidence レイヤを導入し、v0.4.0 で旧系を全削除した。
OPA ポリシーサポート、207件の mutation-hardening テスト、Creusot 形式検証も揃った。

しかし、個々の設計判断を支える**上位原則**が明文化されていない。
新しい contributor や将来の自分が「なぜこの構造なのか」を問うたとき、
ADR-0001 は "何をしたか" を答えるが "なぜそうあるべきか" には答えない。

本 ADR は、全ての設計判断を導出する公理 (principal) を記録する。
個々の ADR はこれらの公理から演繹可能であるべきであり、
矛盾する設計判断が提案された場合はこの ADR を根拠に棄却できる。

## 公理

### P1: 収集・評価・決定の三相分離

```
Adapter (収集) → Control (評価) → Profile (決定)
```

証跡の収集、統制の評価、ゲート判定は独立した相である。
各相は隣接する相の内部表現に依存してはならない。

**導出される制約:**
- Adapter は `ControlStatus` を返さない（評価しない）
- Control は `GateDecision` を返さない（決定しない）
- Profile は `EvidenceGap` を生成しない（収集しない）
- プラットフォーム固有の語彙は収集相に閉じる。`gh-verify-core` は GitHub を知らない
- 新プラットフォーム対応は Adapter の追加のみで完結する

**根拠:** 責務の混在は、変更理由の多重化を引き起こす。
GitHub API の変更、SLSA 要件の改訂、組織ポリシーの変更は
それぞれ独立した変更理由であり、独立した相に閉じるべきである。

### P2: 不確実性は一級市民

`EvidenceState<T>` は4状態を区別する:

| 状態 | 意味 | 例 |
|---|---|---|
| `Complete` | 全データ取得済 | レビュー一覧を完全取得 |
| `Partial` | 一部欠損あり | ページネーション上限到達 |
| `Missing` | 取得失敗 | API エラー、権限不足 |
| `NotApplicable` | そもそも該当しない | release に PR レビューは不要 |

`Result<T, E>` や `Option<T>` では表現できない。
`Missing` は「試みて失敗した」であり、`NotApplicable` は「試みる必要がない」である。
この区別がなければ、Control は `Indeterminate` と `NotApplicable` を正しく出し分けられない。

**導出される制約:**
- `unwrap()` や `expect()` で `EvidenceState` を潰さない
- `Missing` を `Violated` に、`NotApplicable` を `Satisfied` に暗黙変換しない
- 不確実性の解釈は Profile の責務（P1 参照）

### P3: 判定核の形式検証

セキュリティ判定の核となる述語は Creusot + SMT ソルバで正しさが証明される。

```
crates/verif/   — 形式仕様 (#[ensures])
crates/core/    — 実行時実装 (同一ロジック)
```

**導出される制約:**
- 検証対象の述語はプリミティブ型のみ使用する（`String`, `Vec`, `format!` 禁止）
- 実行時実装は検証済み述語と論理的に同値でなければならない
- 同値性は網羅テスト（全入力組合せ）で補強する
- 述語の変更は `devenv tasks run ghverify:verify` の通過を要する
- Control は純粋関数でなければならない（形式検証の前提条件）

**根拠:** 「テストは虫の存在を示せるが、不在を示せない」(Dijkstra)。
四眼原則やコミット署名検証のような安全性不変条件は、
テストでは十分にカバーできない。形式証明が唯一の保証手段である。

## 公理間の関係

```
P1 (三相分離) ─── 構造の公理
├── P2 (不確実性) を要請する: 収集相が不確実性を明示しなければ、評価相は Indeterminate を出せない
└── プラットフォーム中立性を導出する: 収集相がプラットフォーム依存を吸収する

P3 (形式検証) ─── 正しさの公理
├── Control の純粋性を導出する: 副作用のある関数は検証できない
└── P2 を補強する: 不確実性の潰し込みは検証済み述語の前提条件を壊す
```

P1 と P3 は独立した2つの根であり、P2 は両者から要請される。

## 公理の適用例

### 「OPA ポリシーサポートを追加する」
- P1 により、OPA は決定相に配置される → `ControlProfile` trait を実装
- P1 により、OPA エンジンは CLI クレートに配置される → core は regorus に依存しない

### 「新しい Control を追加する」
- P3 により、判定核の述語は `crates/verif/` に仕様を追加する
- P3 により、Control は `&EvidenceBundle` のみを入力とする純粋関数とする
- P1 により、Control は GitHub 型を参照しない

### 「GitLab 対応を追加する」
- P1 により、`crates/cli/src/adapters/gitlab.rs` を追加するだけで済む
- P1 により、Control と Profile は一切変更不要
- P2 により、GitLab API の制約は `EvidenceGap` として表現される

## 改訂条件
- 公理間に矛盾が発見された場合
- 公理から導出不能な設計判断が正当化される場合
- システムのドメインが SDLC 検証から大きく逸脱する場合

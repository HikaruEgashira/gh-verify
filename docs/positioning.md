## 設計方針

- **保管はGitHubに寄せる**: attestation保管、証跡保管等のストレージ機能は自前で持たず、GitHubの既存機能 (Artifact Attestations, Actions Artifacts等) に委譲する
- **GitHubにできない検証を提供する**: gh-verifyの価値は保管ではなく、GitHubが提供しない検査ロジックにある

## gh-verify の独自性

1. **形式検証されたルールロジック** — Creusot + SMTで判定ロジック自体の正しさを数学的に証明。競合にこの層はない
2. **lint的パラダイム** — 大量のルールを決定的・ヒューリスティックに高速実行
3. **仕様マッピング** — PRが事前定義された仕様とマッピング可能かを検知し変更の妥当性を検証
4. **SOCレポートのカスタマイズ生成** — 組織運用に応じたコンプライアンス文書を検査結果から生成 (構想段階)
5. **GitHub外ツール連携** — Jira等との連携を設計に内包。GitHubだけでは完結しない開発プロセス (要件定義、設計レビュー等) の検証を可能にする

## 競合との軸の違い

| 軸 | gh-verify | 競合 |
|----|-----------|------|
| 正しさの保証対象 | ルールロジック自体 (形式検証) | データの改竄防止 (暗号署名) / プロセスの強制 |
| 保管 | GitHubに委譲 (自前で持たない) | 独自プラットフォームに集約 (JFrog Artifactory, Aviator SaaS) |
| 検査スタイル | lint的: 大量ルール高速実行 | ワークフロー自動化 / 証跡収集 |
| カバー範囲 | 設計〜リリースの開発プロセス全体 | PRマージ〜デプロイ中心 |
| 外部ツール連携 | 設計に内包 (Jira等) | GitHub中心 / パートナーエコシステム |
| GitHubとの関係 | GitHubの検証能力を**補完** | GitHubを**置換**または**ラップ** |

## 不足しているもの

| 機能 | 対応状況 | 優先度 |
|------|----------|--------|
| GitHub外ツール連携 (Jira等) | 設計に余地あり・未実装 | 高 |
| SOCレポート生成 | 構想段階 | 高 |
| ダッシュボード / Web UI | なし | 中 |
| パートナーエコシステム | なし | 低 |
